# Phase 04.1: Visual Concept - Research

**Researched:** 2026-03-06
**Domain:** Pixel art visual design, game asset planning, HTML Canvas sprite rendering
**Confidence:** HIGH (core technical facts), MEDIUM (aesthetic recommendations)

---

## Summary

Soggy Moggy is a portrait-format vertical platformer whose character is a stuffed cat, not a real cat. The visual direction needs to communicate "cozy danger" — charm up top, rising peril below. All art is pixel art made in Pixelorama at 64x64 source resolution, displayed at 32x32 in-game (2x downscale, not upscale). Sprites already in progress (beige cat sitting, grey cat paw-up, cloud layers, day/night sky strips) establish a working vocabulary: warm earthy character tones, muted pink/grey clouds, clear sky gradient from daytime blue to night indigo.

The single most important finding: the two existing cat sprites already define the palette anchor. Cat1_beishe (warm tan/grey body, pink ears, dark button eyes) is the stronger candidate for the idle/main sprite. Its color range — dusty tan, warm grey, coral pink ear, teal-blue button eyes — maps cleanly onto a 16-color working palette. All new assets should be built from colors already present in these sprites extended with platform, water, and background tones.

The second important finding: at 64x64 source displayed at 32x32, pixel-perfect rendering requires `ctx.imageSmoothingEnabled = false` set once when the context is created and never overridden. This is a single line of code, but if it is missing the sprites will look blurry at all sizes.

**Primary recommendation:** Use the existing beige cat sprite as the palette seed. Extend it into a 16-color working palette covering background, platforms, water, and HUD. Draw every asset at 64x64, export as PNG, display at 32x32. One style rule above all others: colored outlines (no hard black), hue-shifted shadows.

## Visual Style Approach

### Chosen Style: Warm Pixel Art with Plush-Toy Character

**Rationale:**
- Tooling is already chosen (Pixelorama) and partially used
- Existing sprites commit to a round, soft-body pixel art style
- 64x64 source resolution supports sufficient detail for a plush toy feel without requiring animation frames for MVP
- The "cozy danger" atmosphere reads cleanly in pixel art: soft warm tones dominate the upper areas of the canvas; water and dark sky dominate the lower/flooding regions

**Style rules (the six pillars):**

| Rule | Decision |
|------|----------|
| Resolution | 64x64 source, displayed at 32x32 (drawImage to 32x32 output) |
| Palette | 16 colors, shared across ALL assets — see palette section |
| Outlines | Colored outlines only — no pure black (#000000). Use darkest shade of adjacent color |
| Shading | Hue-shifted cel shading — 2 shades per surface, shadow hue shifts slightly cool |
| View angle | Side-view (character faces right by default) |
| Anti-aliasing | None — every pixel intentional. Jagged diagonals are correct. |

**Reference aesthetic:** The sitting beige cat (Cat1_beishe) already defines the target feel. Match that softness and round silhouette across all game objects.

### What NOT to do

- Do not use pure black (#000000) anywhere except as a temporary placeholder
- Do not use the current neon green `#27ae60` — it reads as debug color, not game art
- Do not add more than 2 highlight/shadow tones per surface at this sprite scale
- Do not use gradients — flat fill + outline is the correct approach at 32x32 display size

## Character Design Notes

### Stuffed Cat at 64x64

**What exists:** Cat1_beishe is already close to the target. It has: round body, pointed ears with pink interior, dark button eyes, short curved tail, visible paw separation, dark outline on a light background. The proportions read as cat-like but soft.

**What makes "stuffed toy" different from "real cat" at small pixel sizes:**

| Visual cue | How to pixel it at 64x64 |
|------------|--------------------------|
| Button eyes | 2x2 or 3x3 pixel solid blocks (teal-blue `#3a7a8a`), with 1px dark pixel as "button center" — not curved eye shapes |
| Seam lines | 1px lighter line running along body center and limb joins — use BG-3 `#ffd4a3` (lightest cat tone) |
| Floppy limbs | Limbs end in rounded blobs with no claws — use 2-3 pixel round caps, not pointed ends |
| Plush texture cue | Single horizontal "fur direction" hatch lines (1px) across body belly — 2-3 lines, not full hatching |
| Rounded silhouette | Avoid sharp angles in body outline entirely — all corners should be 1px steps, never straight right angles |
| Sitting pose | Body mass low, head large relative to body — "plush proportions" (head 40% of total height) |

**Cat1_beishe is the main sprite.** Do not redesign it. Refine it:
1. Check that button eyes are clearly distinct blocks (not smeared)
2. Add 1-2 pixel seam lines at body joints if not present
3. Ensure limb tips are round blobs

**Cat1_grey_Paw-Up is the throw pose candidate.** It shows a raised paw which maps perfectly to the throw mechanic. Keep the grey coloration to distinguish "throwing" from "idle" — it reads as a different energy state.

### Sprite States Required for Phase 5

| State | Source file | Modifications needed |
|-------|-------------|----------------------|
| Idle / falling | Cat1_beishe | Review button eyes and seam lines |
| Throw pose | Cat1_grey_Paw-Up | Verify paw direction (should aim downward for throw) |
| Damage blink | Cat1_beishe | Already handled in code via blink timer — no sprite change needed |

**v2 animation frames (NOT Phase 5 scope):** squash on landing, stretch on ascent, multi-frame jump arc. These are POL-01 in REQUIREMENTS.md and explicitly deferred.

## Background Design Notes

### Strip-Based Scrolling Background

The project uses stacked vertical strips, tiling vertically with camera scroll (Approach 1 from memory). Layer1 already contains day.png (muted blue), night.png (dark indigo), Clouds_bright.png (pink-white cloud shapes), and Clouds_dark.png (grey-blue clouds).

**Strip rendering rule (from codebase review):**
```javascript
// Tile formula from MEMORY.md
bgY = cameraY % stripHeight;
```
Each strip must tile seamlessly at its own height. The top and bottom pixel rows of every background strip must match.

**Layer render order (from MEMORY.md):**
1. Far background (BG-1 sky solid fill — fillRect, not a sprite)
2. Mid background strip (clouds, 60% alpha) — world space
3. Platforms — world space
4. Player — world space
5. Water overlay — world space
6. HUD — screen space (after ctx.restore)

### Background Visual Rules

**Depth separation by saturation and value:**

| Layer | Rule | Reason |
|-------|------|--------|
| Far background | Desaturated, mid-value (BG-1 `#7eb8c9`) | Should recede behind everything |
| Cloud strips | 60% alpha overlay of BG-3 `#ffd4a3` shapes on transparent | Soft, not competing with gameplay |
| Platforms | High saturation (PLAT-1), full alpha | Must read as solid interactive surfaces |
| Player | Warm, full contrast | Always the visual anchor |
| Water | Dark, desaturated blue (WATER-1), 75% alpha | Threatening but not overwhelming |

**Day/night transition:** The existing day.png and night.png strips in Layer1 suggest a planned day/night sky progression. For MVP this can be a simple toggle based on height reached: lower heights = day, higher heights = night. This is achievable with a single background color swap (fillRect BG-1 vs BG-2) with no additional sprite work.

**What NOT to build in backgrounds:**
- No parallax scroll for MVP — the memory note is explicit: Approach 1, no parallax
- No animated weather — that is post-MVP visual polish
- No story-specific environment changes — pure visual atmosphere only

### Cloud Sprites (existing assets)

Clouds_bright.png and Clouds_dark.png are already drawn and sized for Layer1 (480px wide strips). They use simple rounded pixel cloud shapes in pink-white and grey-blue respectively. These are ready to use as-is by loading them as Image objects and rendering with drawImage at 60% alpha.

## Canvas Rendering Notes

### imageSmoothingEnabled

Set once, immediately after context creation. This is already the right place in main.js:

```javascript
// In main.js, after getContext():
const ctx = canvas.getContext('2d', { alpha: false });
ctx.imageSmoothingEnabled = false;  // ADD THIS LINE — prevents sprite blur
```

**Source:** MDN Web Docs (Baseline Widely Available since April 2017). No cross-browser fallback needed for any modern browser.

**CRITICAL TRAP:** `ctx.save()` and `ctx.restore()` do NOT save/restore `imageSmoothingEnabled` in all browsers (behavior is inconsistent). Set it again after any `ctx.restore()` if there is any doubt. Safest approach: set it once at startup and never call ctx.save/restore in the sprite rendering path (the current code already does this correctly — sprites render inside the save/translate/restore block but the property persists).

**Verification:** After adding the line, load a sprite and zoom in — pixels should have hard edges, not fuzzy halos.

### drawImage at 2x downscale (64x64 source → 32x32 display)

```javascript
// Source image is 64x64. Draw it at 32x32 on canvas.
ctx.drawImage(catIdleImage, 0, 0, 64, 64,   // source rect (full sprite)
              Math.floor(player.x),           // dest x — floor to integer
              Math.floor(player.y),           // dest y — floor to integer
              32, 32);                         // dest size matches player.w/h
```

**Why floor():** Non-integer destination coordinates cause sub-pixel blending even with imageSmoothingEnabled = false. The current code already uses `Math.floor()` in `fillRect` calls — keep this convention when switching to `drawImage`.

### Image loading pattern (no framework)

```javascript
// Preload all sprites before the game loop starts
const sprites = {};

function loadSprites(onComplete) {
  const toLoad = {
    catIdle:  'PixelArt/Cats/cat_idle.png',
    catThrow: 'PixelArt/Cats/cat_throw.png',
    // ...
  };
  let pending = Object.keys(toLoad).length;
  for (const [key, src] of Object.entries(toLoad)) {
    const img = new Image();
    img.onload = () => { if (--pending === 0) onComplete(); };
    img.src = src;
    sprites[key] = img;
  }
}

// In main.js:
window.addEventListener('load', () => {
  loadSprites(() => {
    requestAnimationFrame(gameLoop);  // start loop only after all images loaded
  });
});
```

**Source:** Standard Canvas 2D pattern, verified against MDN `drawImage` documentation.

### CSS: image-rendering

Add to index.html or CSS to prevent browser-level upscaling blur if the canvas is ever CSS-scaled:

```css
canvas {
  image-rendering: pixelated;       /* Chrome, Edge, Opera */
  image-rendering: crisp-edges;     /* Firefox */
}
```

This is a belt-and-suspenders measure. The canvas is currently set to 480x640 via JS attributes and not CSS-scaled, so this is LOW priority but correct to add.

## Key Decisions Recommended for the Planner

These are the open choices this research surfaces. The planner should turn each into a concrete task or locked decision.

### Decision 1: Palette lock

**What:** Before any new sprite is drawn, the 16-color palette above should be imported as a custom palette into Pixelorama. All existing .pxo files should be checked against it.

**Recommended action:** Create a Pixelorama palette file (.json or .gpl) with the 16 colors listed in this document. Import it. Check Cat1_beishe against it — the existing colors are close but not exact.

### Decision 2: Platform rendering approach

**What:** Platforms have variable widths (60-100px). Options:
1. Keep `fillRect` with new palette colors (PLAT-1/PLAT-2) — fast, consistent, no sprite work
2. Draw a 16x12 tile sprite and repeat it — more visual richness, requires a tiling loop in renderPlatforms
3. Draw pixel-art end-caps (left 8px, right 8px) and stretch a 1px middle — most realistic look

**Recommended for MVP:** Option 1 with updated palette colors. This changes 4 lines in platforms.js and requires zero new sprites. End-cap sprites are v2 polish.

### Decision 3: Background implementation timing

**What:** The existing day.png, night.png, Clouds_bright.png assets need to be wired into the render loop. This is Phase 5 coding work, but the assets exist.

**Recommended action:** Include background wiring as a Phase 5 task: load the strip images, render them in layer order before platforms, at 60% alpha for cloud layers.

### Decision 4: Heart icon vs Unicode glyph

**What:** Current HUD uses Unicode ♥. For submission quality, pixel-art heart icons read better.

**Recommended:** Draw heart icons only if time allows after Priority 1 assets are complete. 16x16 hearts take approximately 30 minutes each in Pixelorama.

### Decision 5: Single sprite vs spritesheet

**What:** Vanilla JS with `drawImage` works with either individual PNG files or a spritesheet. Spritesheets require an offset/clip calculation per sprite.

**Recommended for one-person project:** Individual PNG files per sprite state. No spritesheet tooling needed. Spritesheet is a v2 optimization if asset count grows beyond 20 sprites.

## Common Pitfalls

### Pitfall 1: Sprite blur on canvas
**What goes wrong:** drawImage renders fuzzy/blurry sprites
**Why it happens:** `imageSmoothingEnabled` defaults to `true`; non-integer coordinates cause sub-pixel blending
**How to avoid:** Set `ctx.imageSmoothingEnabled = false` immediately after `getContext`. Floor all draw coordinates.
**Warning signs:** Sprites look fine in Pixelorama but blurry in-game

### Pitfall 2: Colors that read fine at 64x64 but disappear at 32x32
**What goes wrong:** Fine detail (single-pixel seam lines, small button eye details) becomes invisible at display size
**Why it happens:** 2x downscale merges adjacent pixels — a 1px detail at 64x64 becomes a 0.5px display pixel (sub-pixel, invisible)
**How to avoid:** Test all sprites at 32x32 in a browser context, not just Pixelorama. Any detail that must read must be at least 2px wide at source size.
**Warning signs:** Seam lines and button eye texture disappear when game runs

### Pitfall 3: Player disappears against background
**What goes wrong:** Cat silhouette merges with background when in certain positions
**Why it happens:** Palette chosen without testing against all background states
**How to avoid:** The warm tan (CAT-1 `#b09070`) against sky blue (BG-1 `#7eb8c9`) has good hue contrast. Check specifically against the night sky (BG-2 `#2e3a5c`) — the body shadow (CAT-2 `#7a6555`) is close in value to night sky. Add the lightest cat highlight (CAT-3 `#e8c8a0`) as visible outline pixels in the night context.
**Warning signs:** Cat looks washed out or invisible in upper levels (night sky)

### Pitfall 4: Platform colors competing with water
**What goes wrong:** Crumble platform warning colors (orange/yellow) read as water-adjacent, confusing the player
**Why it happens:** Blue and orange are complementary — they vibrate visually when adjacent
**How to avoid:** Use the earthy brown-orange (PLAT-3 `#c0662a`) and amber (PLAT-4 `#e8a830`) rather than the current bright orange/yellow. These are warm but not saturated enough to compete with the water's blue.
**Warning signs:** Playtesters describe crumble platforms as "looking like they're on fire or flooded"

### Pitfall 5: Forgetting to export PNGs from .pxo files
**What goes wrong:** Code references .png paths but only .pxo files exist in the repo
**Why it happens:** Pixelorama saves in its own format; PNG export is a separate step
**How to avoid:** After every Pixelorama session, export PNG alongside the .pxo. Keep both in the repo.
**Warning signs:** Image.onload never fires, sprites array stays empty, game loop never starts

## Code Examples

### Setting up crisp rendering (verified, MDN source)
```javascript
// Source: https://developer.mozilla.org/en-US/docs/Web/API/CanvasRenderingContext2D/imageSmoothingEnabled
const ctx = canvas.getContext('2d', { alpha: false });
ctx.imageSmoothingEnabled = false;  // set once at startup
```

### Replacing fillRect player with drawImage (pattern)
```javascript
// In renderPlayer(ctx):
// Replace:
//   ctx.fillStyle = '#e74c3c';
//   ctx.fillRect(Math.floor(player.x), Math.floor(player.y), player.w, player.h);
// With:
function renderPlayer(ctx) {
  if (water.iframeTimer > 0 && Math.floor(water.iframeTimer * 5) % 2 === 1) return;
  const img = GameState.throwing ? sprites.catThrow : sprites.catIdle;
  ctx.drawImage(img, 0, 0, 64, 64,
                Math.floor(player.x), Math.floor(player.y),
                player.w, player.h);  // player.w = player.h = 32
}
```

### Tiling a cloud strip (pattern)
```javascript
// In render(), before platforms, inside save/translate block:
function renderBackground(ctx) {
  // Layer 1: sky fill (no sprite needed — solid color)
  ctx.fillStyle = '#7eb8c9';  // BG-1 day sky
  ctx.fillRect(0, GameState.cameraY, 480, 640);

  // Layer 2: cloud strip (tiles vertically with camera)
  if (sprites.cloudsBright) {
    ctx.globalAlpha = 0.6;
    const stripH = sprites.cloudsBright.height;  // 640
    // Tile: draw current strip and the one above/below to cover scroll seams
    const bgY = GameState.cameraY - (GameState.cameraY % stripH);
    ctx.drawImage(sprites.cloudsBright, 0, bgY);
    ctx.drawImage(sprites.cloudsBright, 0, bgY - stripH);
    ctx.globalAlpha = 1.0;  // reset always
  }
}
```
